# /learn [story-id]

เมื่อผู้ใช้เรียกใช้คำสั่ง `/learn <story-id>` (เช่น `/learn 1-2-keycloak-identity-rbac`) ให้ดำเนินการดังนี้:

1. **ค้นหาข้อมูล Story**:
    - อ่านไฟล์ `_bmad-output/planning-artifacts/epics.md` เพื่อหา Story ที่ตรงกับ ID ที่ระบุ
    - ตรวจสอบ Acceptance Criteria และ Goal ของ Story นั้นๆ

2. **วิเคราะห์บริบททางเทคนิค**:
    - ตรวจสอบ `_bmad-output/planning-artifacts/architecture.md` และ `prd.md` หากจำเป็น เพื่อเข้าใจความเชื่อมโยง
    - ตรวจสอบ implementation ปัจจุบันใน codebase (หากมี) เพื่อเปรียบเทียบกับเป้าหมาย

3. **ตอบกลับผู้ใช้ด้วยโครงสร้างดังนี้ (ในภาษาไทย)**:

    ## 📘 ข้อมูลประกอบการเรียนรู้: [ชื่อ Story] (5W Pattern)

    ### 1. Who (ใครเกี่ยวข้อง?)
    - ระบุ User Role หรือ System Component ที่ได้รับผลประโยชน์จาก Story นี้
    - อธิบายว่าใครเป็นคนใช้งานหรือใครที่ระบบต้องไปปฏิสัมพันธ์ด้วย

    ### 2. What (ทำอะไร?)
    - อธิบายเป้าหมายและฟีเจอร์หลักในรูปแบบที่เข้าใจง่าย ทั้งในเชิงธุรกิจและเทคนิค
    - ระบุสิ่งที่กำลังจะถูกสร้างหรือเปลี่ยนแปลง

    ### 3. Where (ทำที่ไหน?)
    - ระบุไฟล์, Module, หรือ Component สำคัญที่เกี่ยวข้องกับการเปลี่ยนแปลงนี้
    - ระบุตำแหน่งในสถาปัตยกรรม (เช่น Frontend, Backend Service, Database)

    ### 4. When (เกิดขึ้นเมื่อไหร่?)
    - ระบุ Trigger Event หรือเงื่อนไขที่ทำให้โค้ดส่วนนี้ทำงาน (เช่น เมื่อได้รับ Event จาก Kafka, เมื่อ User คลิกปุ่ม)
    - อธิบาย Flow หรือลำดับเหตุการณ์ที่เกี่ยวข้อง

    ### 5. Why (ทำไมถึงต้องทำแบบนี้?)
    - อธิบายเหตุผลทางสถาปัตยกรรม (Architectural Decision) และมาตรฐานที่ใช้ (เช่น Security Standards, NestJS Best Practices)
    - ระบุว่าการออกแบบนี้ช่วยแก้ปัญหาอะไร หรือทำไมถึงเลือกใช้แนวทางนี้แทนทางเลือกอื่น

4. **ปิดท้าย**:
    - ถามผู้ใช้ว่าต้องการให้เริ่มขั้นตอน Implementation หรือต้องการให้อธิบายเจาะลึกส่วนไหนเป็นพิเศษหรือไม่
