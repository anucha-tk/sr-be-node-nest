# /learn [story-id]

เมื่อผู้ใช้เรียกใช้คำสั่ง `/learn <story-id>` (เช่น `/learn 1-2-keycloak-identity-rbac`) ให้ดำเนินการดังนี้:

1. **ค้นหาข้อมูล Story**:
    - อ่านไฟล์ `_bmad-output/planning-artifacts/epics.md` เพื่อหา Story ที่ตรงกับ ID ที่ระบุ
    - ตรวจสอบ Acceptance Criteria และ Goal ของ Story นั้นๆ

2. **วิเคราะห์บริบททางเทคนิค**:
    - ตรวจสอบ `_bmad-output/planning-artifacts/architecture.md` และ `prd.md` หากจำเป็น เพื่อเข้าใจความเชื่อมโยง
    - ตรวจสอบ implementation ปัจจุบันใน codebase (หากมี) เพื่อเปรียบเทียบกับเป้าหมาย

3. **ตอบกลับผู้ใช้ด้วยโครงสร้างดังนี้ (ในภาษาไทย)**:

    ## 📘 ข้อมูลประกอบการเรียนรู้: [ชื่อ Story]

    ### 1. เรากำลังทำอะไรอยู่? (What)
    - อธิบายเป้าหมายของ Story นี้ในรูปแบบที่เข้าใจง่าย ทั้งในเชิงธุรกิจและเทคนิค
    - ระบุฟีเจอร์หรือความสามารถใหม่ที่กำลังจะเพิ่มเข้ามา

    ### 2. ทำไมถึงเขียนแบบนี้? (Why)
    - อธิบายเหตุผลทางสถาปัตยกรรม (Architectural Decision) เช่น การเลือกใช้ Guard, Interceptor, หรือ Pattern เฉพาะทาง
    - ระบุว่าการออกแบบนี้ช่วยแก้ปัญหาอะไร หรือทำตามมาตรฐานใด (เช่น NestJS Best Practices, Security Standards)

    ### 3. สิ่งที่เราได้รับคืออะไร? (Outcome)
    - ระบุผลลัพธ์ที่เป็นรูปธรรม (Technical Deliverables) เช่น API Endpoints, Database Schemas, Security Layers
    - อธิบายคุณค่า (Value) ที่ระบบได้รับ เช่น ความปลอดภัยที่สูงขึ้น, ประสิทธิภาพที่ดีขึ้น หรือ Code ที่บำรุงรักษาง่ายขึ้น

4. **ปิดท้าย**:
    - ให้คำแนะนำสั้นๆ เกี่ยวกับไฟล์สำคัญที่ต้องเกี่ยวข้องใน Story นี้
    - ถามผู้ใช้ว่าต้องการให้เริ่มขั้นตอน Implementation หรือต้องการให้อธิบายเจาะลึกส่วนไหนเป็นพิเศษหรือไม่
